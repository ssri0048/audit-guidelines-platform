## ส่วนที่ 1 — สำหรับผู้ตรวจสอบ (ผู้ใช้เว็บ)

### เปิดใช้งาน
เว็บเป็นแบบ **อ่านอย่างเดียว** ไม่ต้อง login — เปิดจาก GitHub Pages ของ repo (path `/app/`)

### ค้นหา
พิมพ์คำถามธรรมดาได้เลย ระบบตีความประเภทองค์กรจากคำถามอัตโนมัติ เช่น
- "จัดซื้อจัดจ้าง**ของรัฐวิสาหกิจ**" → กรองบริบทรัฐวิสาหกิจให้เอง (ความเสี่ยงสากลแสดงร่วมเสมอ)
- หรือกดชิปประเภทองค์กรเอง: ราชการ / รัฐวิสาหกิจ / บริษัทจดทะเบียน / เอกชน / SME / ท้องถิ่น / สหกรณ์-ชุมชน
- ไม่ระบุประเภท → เห็นความรู้สากลก่อน กดปุ่ม "แสดงทั้งหมด" เพื่อขยาย

### อ่านป้ายให้เป็น (หัวใจของความเชื่อถือ)
| ป้าย | ความหมาย |
|---|---|
| `L1 · v2.x` + อนุมัติแล้ว | ผ่านการอ่านแหล่งทางการ ≥2 แหล่ง + quality gates + มนุษย์ merge |
| 📖 อ่านจากแหล่งจริง | เนื้อหาข้อนี้สกัดจากเอกสารที่ระบบเปิดอ่านจริง |
| 🧭 อิงแบบแผนตัวอย่างจริง | เขียนตามโครงจากคลังแนวการตรวจของผู้ตรวจจริง (exemplar) |
| ⚗️ สังเคราะห์เชิงวิชาชีพ | สังเคราะห์จากหลักวิชาชีพ โดยยึดมาตรฐานที่ verify เวอร์ชันแล้ว |
| ชิปเขียว ✓ ปัจจุบัน | มาตรฐานยืนยันเวอร์ชันจากหน้า official แล้ว |
| ชิปเหลือง ⏳/✎ | กำลังจะมีฉบับใหม่ / มีฉบับแก้ไขต้องอ้างคู่ |
| ชิปแดง ✗ | ถูกยกเลิก — ระบบบอกฉบับแทนที่ให้ |
| ◌ ต่อท้าย | อยู่ในทะเบียนแต่ยังไม่ verify จากหน้า official (มีธง+เส้นตายกำกับ) |
| 🏢 ม่วงเส้นประ | มาตรฐานภายในองค์กร — พักไว้รอเชื่อมฐานข้อมูล ไม่ใช่งานค้าง |

### ตรวจย้อนแหล่งที่มา
ทุกหัวข้อมีแผง "แหล่งอ้างอิงพร้อมหลักฐานที่อ่านจริง" — ลิงก์ + ข้อความคัดจากเอกสาร + วันที่เก็บ + คะแนนความน่าเชื่อถือ กดตรวจเองได้ทุกรายการ

### ขอหัวข้อใหม่ / แจ้งข้อมูลคลาดเคลื่อน
ปุ่มท้ายหน้าเว็บ → เปิด GitHub Issue (ไม่ต้องมีบัญชีพิเศษ แค่บัญชี GitHub ธรรมดา)

## ส่วนที่ 2 — สำหรับผู้ดูแลระบบ (admin)

### หลักการเดียวที่ครอบทุกอย่าง
**ประตูเขียนมีบานเดียว: Pull Request** — main ถูกล็อค (Require PR + CI must pass + no bypass) ไม่มีใครรวมทั้ง AI เขียนความรู้เข้าระบบได้โดยไม่ผ่านเกตอัตโนมัติ + การ merge ของคุณ

### วงจรความรู้ (เพิ่ม/แก้หัวข้อ)
1. รับคำขอ (Issue) หรือเปิดเอง
2. Research ใน Cowork ตาม `skills/deep-research-protocol/SKILL.md` — บังคับขั้น 1.5 **Reading Manifest**: list มาตรฐานทุกตัวที่จะอ้าง → ตัวไหนไม่มีทะเบียนต้อง**อ่านหน้า official เข้าทะเบียนก่อน**เขียนความรู้ (CI จะ block ถ้าฝืน — เกต RATCHET)
3. เปิด branch → PR → CI รัน 9 เกต (schema/dedup/แหล่งอ้างอิง/ปีอนาคต/เวอร์ชันมาตรฐาน/ไทย/coverage/internal-lock/hash)
4. คุณอ่าน diff → merge = อนุมัติ

### ธงและเส้นตาย (ระบบทวงเอง)
- ทุกความไม่แน่ใจถูกบันทึกเป็นธง (exception_code + open_question + SLA 60 วัน) — เกิน SLA = CI block จนกว่าจะปิด
- **queue-watchdog** อัพเดท issue "🏳️ Verification Queue" ทุกวันจันทร์เช้าอัตโนมัติ
- กติกา Boy Scout: ทุก batch ปิดธงเก่า ≥2 ก่อนเปิดใหม่ (วิธีปิดดู `skills/verification-resolver/SKILL.md` มี Publisher Lookup Table)

### Pipeline ร่างอัตโนมัติ (ตั้งไว้รอ)
ใส่ Secret `ANTHROPIC_API_KEY` → ติด label `research-request` ให้ issue → ได้ draft PR (L3) + Reading Manifest — **draft ห้ามเข้า store ตรง** ต้อง verify ใน Cowork แล้วยกเป็น L1 ก่อน

### มาตรฐานภายในองค์กร (ตั้งไว้รอ)
`data/platform_flags.json → internal_doc_enabled: false` (fail-closed) — เปิดใช้ = แก้ไฟล์นี้ผ่าน PR เมื่อเชื่อม connector ฐานข้อมูลองค์กรแล้ว | กติกาถาวร: เอกสารภายในไม่นับเป็นแหล่ง L1, ห้าม excerpt ลง repo สาธารณะ, ความรู้ที่ยืนบนมันคือ L2 | รายละเอียด `docs/INTERNAL_KNOWLEDGE_PATH.md`

### เพิ่ม admin คนที่สอง
Settings → Collaborators → เชิญ | จากนั้นเปิด "Require approvals ≥1" ใน branch protection กลับมา (ตอนนี้ปิดเพราะ admin คนเดียว approve PR ตัวเองไม่ได้)

### งานตามรอบ
| รอบ | งาน |
|---|---|
| ทุกจันทร์ | อ่าน issue Verification Queue (อัตโนมัติ) |
| ก่อน 11-12 ก.ย. 2026 | ปิดธง SLA ชุดแรก: IEC×4, FIDIC, TFRS, IEEE, glossary 11 คำ, กค ว23, PDP2024 ฯลฯ |
| ทุกปีงบประมาณ | ITA ออกเกณฑ์ใหม่รายปี — re-verify family `th-ita` |
| ~ทุก 6 เดือน | รายการ verified เก่าเกิน 183 วันจะขึ้นสถานะ STALE ในคิวเอง |

## ส่วนที่ 3 — สิ่งที่ระบบสัญญา (และไม่สัญญา)

**สัญญา:** ทุกความรู้ L1 ตรวจย้อนได้ถึงเอกสารทางการพร้อมข้อความคัดจริง | ทุกมาตรฐานที่ถูกอ้างมีทะเบียนติดตามเวอร์ชัน (ครบ 100% ตั้งแต่ 2026-07-14) | ทุกความไม่รู้ถูกประกาศเป็นธงพร้อมเส้นตาย ไม่มีการเดาเงียบๆ
