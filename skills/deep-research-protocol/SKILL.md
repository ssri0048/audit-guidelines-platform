# Deep Research Protocol — ระเบียบวิธีการได้มาซึ่งความรู้ที่ตรวจย้อนได้

## Skill Identity
- **Name**: deep-research-protocol
- **Version**: 1.0.0
- **บังคับใช้กับ**: SKL001 (Knowledge Harvester), SKL002 (Standards Validator), SKL003 (Thailand Applicability Checker) และงาน re-verify ทุกครั้ง
- **หลักการสูงสุด**: *"ไม่เชื่อ AI — เชื่อหลักฐานที่ตรวจซ้ำได้"* ทุกข้ออ้างต้องชี้กลับไปยังข้อความจริงที่อ่านมาจากแหล่ง official ได้

## ขั้น 0 — Zero-Base Sweep (บังคับก่อนสร้าง/ขยายชั้นความรู้ใดๆ)

> บทเรียนจาก failure pattern ที่เกิดซ้ำ 4 ครั้ง: ทุกชั้นที่ derive จากชั้นก่อนหน้า จะรับจุดบอดเป็นมรดก

1. **นิยามจักรวาลจาก research สด** — ห้ามตอบจากความจำหรือ derive จากของที่มี: ถามว่า "วงการนี้มีผู้เล่น/เอกสารอะไรทั้งหมดในโลก" แล้วค้นใหม่ทุกครั้ง
2. **Diff กับ `data/profession_map.json` + ของที่มี** — รายการตก = ธง SLA ทันที
3. **ทุกรายการต้องมี `inclusion_rationale`** — เข้าเพราะอะไร/ไม่เข้าเพราะอะไร โดยเฉพาะเกณฑ์ไทย: อะไรที่ผู้ตรวจไทยควรใช้/ถูกบังคับ ต้องเข้า พร้อมสายโยงว่าเข้าไทยทางไหน (กฎหมาย/หน่วยกำกับ/สมาคมวิชาชีพ)
4. อัพเดท profession_map ทุกครั้งที่พบ body ใหม่

## กติกา Exemplar เพิ่มเติม (content_role)

- ทุก exemplar ต้องมี `content_role`: **NORM** (เกณฑ์—ต้อง verify เต็ม: เวอร์ชัน+lifecycle+ไทย) / **PATTERN** (แบบแผนการเขียนเท่านั้น—ห้ามยกสาระไปใช้ตรง ไม่ต้องเช็คไทย) / **BOTH**
- **Dual-entry**: content_role BOTH ต้องมี `registry_family` ชี้รายการในทะเบียนมาตรฐาน (เข้าทะเบียนก่อน → ค่อยเข้าคลัง)
- **Legal currency**: เอกสารที่ exemplar อ้างภายใน ต้องถูกเทียบทะเบียน — อ้างของ WITHDRAWN → ลดเป็น PATTERN_ONLY

## ขั้นตอนบังคับ 6 ขั้น

### ขั้น 1 — เลือกเอกสารให้ตรงเรื่อง ไม่ใช่แค่ตรงตระกูล
รู้จักโครงสร้างเอกสารของผู้ออกมาตรฐานก่อนหยิบ:
- **INTOSAI (IFPP)**: INTOSAI-P (หลักการ) → ISSAI (มาตรฐานบังคับ) → GUID (แนวปฏิบัติรายหัวข้อ, ตระกูล 5000 = เฉพาะเรื่อง) — ทะเบียนอยู่ที่ issai.org
- **IIA**: GIAS 2024 (5 Domains, Standard เลขทศนิยม เช่น 14.1) แทนที่ IPPF เดิม (เลข 4 หลัก เช่น 2130) — เจอเลข 4 หลักให้สงสัยว่าเป็นระบบเก่า
- **ISO/IEC**: เลขมาตรฐาน + ปี edition (27001:2022) + Amd/Cor แยกชิ้น
- **ไทย**: พ.ร.บ. (ราชกิจจานุเบกษา = ต้นฉบับ, กฤษฎีกาฉบับ update = ตัวบทปัจจุบัน) → กฎกระทรวง → ระเบียบ → ประกาศหน่วยกำกับ (กกพ., สกมช.)

**ห้าม**: หยิบชื่อองค์กร + เลขอะไรก็ได้มาประกอบกัน ต้องยืนยันว่าเอกสารเลขนั้น "ว่าด้วยเรื่องนั้นจริง"

### ขั้น 2 — อ่านไส้ในจริง + เก็บหลักฐาน 4 อย่าง
ก่อนอ้างแหล่งใดเป็น source ต้องมีครบ:
1. `url` — ลิงก์หน้า/ไฟล์ official ที่เปิดได้จริง
2. `excerpt` — ข้อความจริงที่คัดจากเนื้อหา (พิสูจน์ว่าอ่านแล้ว ไม่ใช่อ้างลอย)
3. `retrieved_at` — วันที่อ่าน
4. `credibility_score` — ตาม Authority Scoring (L0 official ≥90)

### ขั้น 3 — ตรวจวงจรชีวิตเวอร์ชัน (ป้อน standards_registry.json)
เปิดหน้า lifecycle ของผู้ออกมาตรฐานแล้วจดสถานะจริง — สถานะที่เป็นไปได้มี 8 แบบ (นิยามใน registry):
`CURRENT / CONFIRMED / TO_BE_REVISED / TRANSITION / WITHDRAWN / AMENDED / PARTIAL_REPEAL / PINNED_BY_THAI_LAW`

วิธีอ่านต่อแหล่ง:
- **iso.org**: ดูส่วน "Life cycle" — stage code บอกตรงๆ (60.60 published, 90.93 confirmed, 90.92 to be revised, 95.99 withdrawn) และดูหัวข้อ "Amendments" ว่ามีของที่ต้องอ้างคู่ไหม
- **theiia.org / nist.gov / coso.org**: หา effective date และประกาศ supersede
- **กฎหมายไทย**: ตามหา "(ฉบับที่ n)" เสมอ + ดูมาตรายกเลิก (เช่น มาตรา 3 พ.ร.บ.จัดซื้อ 2560 ยกเลิกบทบัญญัติพัสดุในกฎหมายอื่น) + ยึดฉบับ update ของ krisdika เป็นตัวบทปัจจุบัน

ผลการตรวจต้องอัพเดทลง `data/standards_registry.json` (พร้อม evidence_url + last_verified) มากับ PR เดียวกัน

### ขั้น 4 — พิสูจน์ Thailand Applicability ด้วยหลักฐาน ไม่ใช่คำประกาศ
`thailand_applicable: true` ต้องมีหลักฐานอย่างน้อยหนึ่งแบบ:
- กฎหมาย/ระเบียบไทยบังคับเรื่องนั้นตรงๆ (อ้างฉบับ+มาตรา)
- หน่วยงานไทยรับมาตรฐานนั้นอย่างเป็นทางการ (เช่น สตง. เป็นสมาชิก INTOSAI และเผยแพร่หน้ามาตรฐาน INTOSAI บน audit.go.th)
- หน่วยกำกับประกาศอ้างมาตรฐานฉบับเจาะจง (= PINNED_BY_THAI_LAW ในทะเบียน)

### ขั้น 5 — แยก "บังคับ" กับ "แนวปฏิบัติ" แล้วบันทึกข้อจำกัด
ทุก source ระบุบทบาท: กฎหมายบังคับ (พ.ร.บ./ระเบียบ) vs มาตรฐานสมัครใจ vs guidance (GUID, best practice) — ข้อจำกัดต้องเขียนลง notes/excerpt เช่น "GUID 5280 เป็นแนวปฏิบัติ ไม่ใช่ข้อบังคับ"

### ขั้น 6 — ซื่อสัตย์เมื่ออ่านไม่ได้ + ตรวจ citation เดิม
- แหล่งเปิดไม่ได้/ไฟล์สแกน → ลดระดับคำอ้างเหลือเท่าที่พิสูจน์ได้ (เช่น "ยืนยันการมีอยู่+วันประกาศ") และเขียนบอกใน excerpt
- ทุกครั้งที่ re-verify ให้ตรวจ citation เก่าว่า "ผิดฝาผิดตัว" หรือไม่ ก่อนคัดลอกต่อ

## ขั้น 7 — Exemplar Grounding + หลายภาษา + Glossary (Provenance & Coverage)

1. **ก่อน gen procedures**: โหลด exemplars ของ domain จาก `data/exemplars/` ≥3 ชิ้น (ภาษาไหนก็ได้ — SAI/หน่วยตรวจรัฐ/องค์กรวิชาชีพทั่วโลก ดูเป้าหมายใน source_universe.json) — ไม่มีพอ → เก็บ exemplar ก่อน หรือ label `PROFESSIONAL_SYNTHESIS` พร้อมเหตุผล
2. **ทุก procedure ติด** `derivation` + `exemplar_refs[]` (ถ้า grounded)
3. **หลายภาษา**: ต้นฉบับอยู่คู่คำแปลเสมอ (excerpt_original + excerpt_th) — แปลผ่านกติกา `data/glossary.json` 3 ชั้น (ศัพท์บัญญัติ → แปล+วงเล็บ → ทับศัพท์มาตรฐาน); เจอศัพท์ใหม่ที่ไม่อยู่ใน glossary → เพิ่ม entry (UNVERIFIED+SLA) มากับ PR เดียวกัน
4. **org_applicability**: ทุก topic/risk ประกาศป้ายชัด (ไม่มีป้าย = UNIVERSAL ต้องตั้งใจเลือก ไม่ใช่ลืม); กฎหมายที่ scope แคบใส่ `org_scope` ราย ref; matching = Match ANY
5. **Source Universe**: แตะ domain ไหน → อัพเดท coverage ของ publishers ที่เกี่ยวใน `data/source_universe.json` ไปพร้อมกัน (harvest คู่ขนาน ไม่เปิดรอบแยก)

## ขั้น 1.5 — Reading Manifest (บังคับ: sources-first ก่อนประกอบความรู้)

> เพิ่ม 2026-07-13 หลังพบว่า "หน่วยของงานเป็นหัวข้อ" ทำให้มาตรฐานรองหลุดเรดาร์ 42 รายการ

**ก่อนเขียนความรู้แม้แต่บรรทัดเดียว** ต้องทำ manifest:
1. List สตริงมาตรฐาน/กฎหมาย **ทุกตัว** ที่จะปรากฏที่ใดก็ตามในหัวข้อ (applicable_standards + thai_law_refs + std_refs ทุก procedure)
2. เช็คทุกสตริงกับ standards_registry.json (`famFor`)
3. ตัวที่ไม่มี family: **อ่านแหล่ง official เข้าทะเบียนก่อน** (ขั้น 2-3 ปกติ) หรือติดธง exception_code+SLA — ห้ามเงียบ
4. แล้วจึงประกอบความรู้ — ทุกการอ้างอิงจะยืนบนทะเบียนที่มีหลักฐาน ไม่ใช่ความจำ

**Ratchet (บังคับโดย CI):** `G5_COVERAGE_NEW` = ERROR — การอ้างอิงใหม่ที่ไม่มี family เข้าระบบไม่ได้
ของเก่า 42 รายการอยู่ใน `data/coverage_baseline.json` (กองมีแต่ลด: research แล้ว**ลบออกจาก baseline**ใน PR เดียวกัน)

**กติกา normalize สตริง:** ตอน research gap เข้าทะเบียน ให้รวมสตริงที่หมายถึงสิ่งเดียวกัน (เช่น "PDPA พ.ศ. 2562" / "พ.ร.บ.คุ้มครองข้อมูลส่วนบุคคล พ.ศ. 2562 (PDPA)") เป็นชื่อมาตรฐานเดียว แล้วแก้ทุก topic ที่อ้างให้ใช้ชื่อเดียวกัน — ลด fragmentation

## กติกาจำนวน Control Failure (มติ 2026-07-14)

จำนวน CF ต่อความเสี่ยง = **ตามเหตุผลเชิงวิชาชีพ ไม่ใช่โควตา**
- โดยธรรมชาติความเสี่ยงหนึ่งมักมีจุดควบคุมล้มเหลวได้หลายจุด (สอง สาม หรือมากกว่า) — ระบุครบเท่าที่มีจริง
- **ห้ามยัดเพิ่มเพื่อให้ครบจำนวน** — หนึ่งข้อก็ถูกต้องถ้าความจริงมีเท่านั้น
- วิธีตรวจ + หลักฐาน ล้อราย CF ของมันเอง ห้ามชุดเดียวหว่านทุก CF
- **จงใจไม่มีเกตนับจำนวน CF** (เกตเชิงปริมาณสร้างแรงจูงใจยัดไส้) — คุณภาพส่วนนี้จบที่ human review ใน PR
